# MEM_011 息屏保活加强

> 补充 MEM_001，记录 v1.1.1.2 新增的两个保活措施。

## v1.1.1.2 发现的缺口

### 缺口 1：loopHandler.postDelayed 套在 uiHandler.post 内部

原代码结构：
```kotlin
uiHandler.post {
    // ... 处理数据、更新 UI
    loopHandler.postDelayed({ runLoop() }, 2000)  // ← 在主线程消息队列内调度
}
```

Android Doze 模式挂起主线程消息队列后，`uiHandler.post {}` 体内的代码根本不执行，`loopHandler.postDelayed` 永远不会被调用，下一轮 `runLoop()` 无法调度——定时链断裂。

**修复**：`loopHandler.postDelayed` 移到 `uiHandler.post` **之前**，从后台线程直接调度。

### 缺口 2：WakeLock 超时版

原代码 `acquire(10000)` 10 秒超时。Doze 下网络延迟大时可能超时释放。

**修复**：改为 `acquire()` 无超时版，手动 `release()`。

### 缺口 3：无 WifiLock

息屏后 WiFi 可能进入低功耗模式，网络请求超时。

**修复**：新增 `WIFI_MODE_FULL_HIGH_PERF` WifiLock，在 `onStartCommand` 中 acquire，`onDestroy` 中 release。

## 完整保活三件套

| 机制 | 用途 | 注意事项 |
|------|------|---------|
| Foreground Service | 进程不被杀 | 必须带通知栏 |
| PARTIAL_WAKE_LOCK | CPU 不休眠 | 无超时版 + 每轮检查 isHeld |
| WIFI_MODE_FULL_HIGH_PERF | WiFi 不休眠 | onDestroy 释放 |

## 仍可能失效的场景

部分国产 ROM（小米 MIUI/澎湃、华为 HarmonyOS、OPPO ColorOS）有额外的后台限制，即使三件套齐全也可能被杀。用户需手动将 App 加入：
- 小米：安全中心 → 应用管理 → StockSpeaker → 省电策略 → 无限制
- 华为：设置 → 应用 → 应用启动管理 → 手动管理（全部开启）
